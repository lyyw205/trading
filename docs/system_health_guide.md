# 시스템 헬스 모니터링 가이드

## 개요

헬스 페이지는 **2-tier 자동 새로고침** 구조입니다.

| 구분 | 주기 | DB 쿼리 | 용도 |
|------|------|---------|------|
| **경량 (Light)** | 5초 | 0건 | CPU, 메모리, DB풀, 웹소켓 등 실시간 지표 |
| **중량 (Heavy)** | 60초 | 5~6건 | 느린 쿼리, Dead Tuple, 거래 활동, 캔들 통계 |

탭을 벗어나면 자동 일시 중지되고, 돌아오면 즉시 재개됩니다.

---

## 1. KPI 카드 (상단 6개)

### CPU 사용률
- **무엇**: 서버 전체 CPU 점유율 (psutil 기반)
- **정상 범위**: 0~70%
- **주의(노란색)**: 70% 이상 — 트레이딩 루프가 많거나 캔들 집계 중일 수 있음
- **위험(빨간색)**: 90% 이상 — Discord 알림 발생
- **활용**: 지속적 90%+ → 활성 계정 수 줄이거나 스캔 인터벌 늘리기

### 메모리 사용률
- **무엇**: 서버 전체 RAM 점유율
- **정상 범위**: 0~70%
- **위험**: 90% 이상 — Discord 알림 발생, OOM Kill 위험
- **활용**: RSS(프로세스 실제 사용량)와 함께 봐야 함. RSS가 낮은데 전체 메모리가 높으면 다른 프로세스 문제

### DB 풀 (사용 중 / 전체)
- **무엇**: SQLAlchemy 커넥션 풀에서 현재 체크아웃된 연결 수 / 풀 전체 크기
- **예시**: `3 / 15` → 15개 중 3개 사용 중
- **위험**: 비율 90% 이상 (예: 14/15) → Discord 알림. 쿼리가 느리거나 연결이 반환되지 않는 상태
- **활용**: 평소 수치를 기억해두고, 갑자기 치솟으면 느린 쿼리 테이블과 함께 확인

### 활성 계정
- **무엇**: 현재 트레이딩 엔진에서 돌고 있는 계정(AccountTrader) 수
- **활용**: 설정한 활성 계정 수와 일치하는지 확인. 숫자가 줄어들면 서킷브레이커가 발동해서 자동 비활성화된 계정이 있을 수 있음

### 웹소켓
- **무엇**: Binance WebSocket 연결 상태 + 구독 수
- **정상**: "정상" (초록색) + 구독 수 표시
- **불량**: "불량" (빨간색) — 바이낸스와의 실시간 데이터 연결 끊김
- **활용**: 불량 시 캔들 데이터가 지연됨. 보통 자동 재접속되지만, 지속되면 서버 재시작 필요

### API 에러율
- **무엇**: 앱 시작 이후 전체 HTTP 요청 중 4xx/5xx 응답 비율
- **정상**: 1% 미만
- **주의**: 1~5% — 클라이언트 오류(404 등) 포함일 수 있음
- **위험**: 10% 이상 → Discord 알림
- **활용**: 에러율이 높으면 아래 API 메트릭 테이블에서 어떤 엔드포인트가 문제인지 확인

---

## 2. 서버 리소스 (왼쪽 패널)

### 게이지 3개: CPU / 메모리 / 디스크
- 시각적 바 차트로 사용률 표시
- 색상 기준: **초록**(~70%) → **노란색**(70~90%) → **빨간색**(90%+)
- **디스크**: 로그파일, 캔들 데이터 누적 시 증가. 90% 넘으면 오래된 로그/캔들 정리 필요

### 하단 3칸: RSS / 스레드 / 업타임

| 항목 | 설명 | 활용 |
|------|------|------|
| **RSS** | 이 프로세스가 실제 사용 중인 메모리 (MB) | 지속적으로 증가하면 메모리 릭 의심 |
| **스레드** | 프로세스 내 스레드 수 | Binance API 호출용 ThreadPoolExecutor 포함. 비정상적 증가 주의 |
| **업타임** | 서버 프로세스 가동 시간 | 최근 재시작 여부 확인. 예상치 않은 리셋이면 OOM Kill 등 원인 조사 |

---

## 3. API 메트릭 (오른쪽 패널)

앱 시작 이후 누적된 인메모리 통계입니다 (재시작 시 초기화).

| 컬럼 | 설명 |
|------|------|
| **엔드포인트** | `GET /api/...` 형태. 숫자/UUID는 `{id}`로 통합 |
| **호출** | 총 호출 횟수 |
| **평균ms** | 평균 응답 시간 |
| **에러율** | 해당 엔드포인트의 4xx+5xx 비율 |

**활용법**:
- **평균ms가 500ms+인 엔드포인트** → 쿼리 최적화 또는 캐시 도입 검토
- **에러율 5%+ (빨간색)** → 해당 API의 로그를 확인
- **호출 수가 비정상적으로 많은 엔드포인트** → 프론트엔드 폴링 주기가 너무 짧거나, 불필요한 반복 호출

---

## 4. 트레이딩 활동 (왼쪽)

### 주문 수 / 체결 수 (최근 1시간)
- **주문 수**: 지난 1시간 동안 생성/업데이트된 주문 총 수
- **체결 수**: 그 중 `FILLED` 상태인 것
- **활용**:
  - 체결/주문 비율이 낮으면 → 주문이 체결되지 않고 대기 중이거나 취소된 것
  - 둘 다 0이면 → 매매가 전혀 없는 상태. BuyPause 상태이거나 전략 조건 미충족
  - 평소 대비 급증 → 시장 변동성 큰 구간, 전략 파라미터 점검

---

## 5. 에러 추이 (오른쪽)

### 최근 6시간 시간대별 ERROR 로그 바 차트
- `persistent_logs` 테이블에서 `level = 'ERROR'`인 로그를 시간 단위로 집계
- **활용**:
  - 특정 시간대에 에러 급증 → 그 시간의 로그를 직접 확인
  - 꾸준히 에러가 발생하면 → 반복적인 API 호출 실패, 바이낸스 연결 문제 등 구조적 원인
  - 바가 모두 0이면 정상

## 6. DB 성능 (전체 너비)

### 커넥션 풀 상세 (5개 필)

| 필 | 설명 | 주의 기준 |
|----|------|----------|
| **풀 크기** | 설정된 최대 커넥션 수 (기본 15) | — |
| **대기 중** | 풀에 반환되어 재사용 가능한 연결 | 0이면 모두 사용 중 |
| **사용 중** | 현재 쿼리 실행 중인 연결 | 풀 크기의 80% 이상이면 노란색 |
| **오버플로** | 풀 초과로 임시 생성된 연결 | 0보다 크면 노란색 — 풀 크기 증가 검토 |
| **활성 연결** | PostgreSQL `pg_stat_activity` 기준 실제 DB 연결 수 | 풀 수치와 큰 차이 나면 외부 연결 존재 |

### 느린 쿼리 (평균 > 200ms)
- PostgreSQL `pg_stat_statements`에서 평균 실행 시간 상위 5개
- **1000ms 이상은 빨간색**으로 표시
- **활용**: 반복 호출되는 느린 쿼리 → 인덱스 추가, 쿼리 최적화, 또는 캐시 도입

### Dead Tuples (상위 10개)
- `pg_stat_user_tables`에서 VACUUM 안 된 죽은 행 수
- **Dead**: DELETE/UPDATE 후 아직 정리되지 않은 행
- **Live**: 현재 유효한 행
- **비율**: Dead/Live — 높을수록 테이블 비대화
- **10,000개 이상은 빨간색**
- **활용**: 비율이 높으면 → `VACUUM ANALYZE 테이블명` 수동 실행 또는 autovacuum 설정 점검

## 7. 캔들 저장소

| 인터벌 | 설명 |
|--------|------|
| 1분 | 바이낸스 WebSocket에서 실시간 수신 |
| 5분 / 1시간 / 1일 | CandleAggregator가 6시간 주기로 1분 캔들 압축 |

- 심볼별 레코드 수 표시
- **활용**: 특정 심볼의 레코드가 비정상적으로 적으면 → 웹소켓 구독 누락 또는 backfill 실패

---

## 8. 알림 배너 & Discord 알림

페이지 상단에 빨간/노란 칩으로 표시되며, 동시에 Discord로 전송됩니다.

| 알림 조건 | 임계값 | 쿨다운 |
|-----------|--------|--------|
| CPU 사용률 | 90% | 5분 |
| 메모리 사용률 | 90% | 5분 |
| DB 풀 점유율 | 90% (사용중/전체) | 5분 |
| API 에러율 | 10% | 5분 |

쿨다운이 있어서 같은 알림이 5분 내 반복 전송되지 않습니다.

---

## 일상적 모니터링 체크리스트

1. **페이지 열었을 때 알림 배너 없는지** 확인
2. **KPI 6개** 훑어보기 — 빨간색/노란색 없으면 정상
3. **활성 계정 수**가 예상과 일치하는지
4. **웹소켓 "정상"** 확인
5. 하루 1회 **느린 쿼리**와 **Dead Tuples** 점검
6. **에러 추이**가 우상향이면 원인 조사
