## 1. ✏️ 프로젝트 전체 개요

이 프로젝트는 **바이낸스(Binance) 거래소에서 암호화폐를 자동으로 매매하는 봇**입니다.
기존 `btc-staking-bot`(단일 계정)을 확장하여, **여러 계정을 동시에 운영**할 수 있도록 설계되었습니다.
BTCUSDT 외에 ETHUSDT 등 다양한 심볼을 지원합니다.

### ✏️ 핵심 특징
- **멀티 계정**: 여러 바이낸스 계정을 독립적으로 운영. 각 계정마다 별도의 API 키, 콤보 설정, 매매 루프를 가짐
- **콤보 기반 전략**: Buy Logic + Sell Logic을 분리하여 조합하는 플러그인 방식. 하나의 계정에 여러 콤보를 동시 운영 가능
- **✏️ 로컬 DB 인증**: 이메일/비밀번호 기반 로컬 인증 (bcrypt). 계정별 접근 권한 제어
- **실시간 대시보드**: 계정별 차트, 로트(매수 묶음) 현황, 전략 파라미터 실시간 조정
- **장애 격리**: 서킷 브레이커(5회 연속 실패 시 자동 비활성화), 지수 백오프, API 속도 제한
- **🆕 실시간 캔들 수집**: WebSocket 기반 1분봉 수신 + 다중 타임프레임(5m/1h/1d) 집계
- **🆕 Buy Pause**: 잔고 부족 시 매수 자동 일시정지 (ACTIVE → THROTTLED → PAUSED 상태 기계)

### ✏️ 기술 스택
```
Python 3.12+ / FastAPI / SQLAlchemy 2.0 (async) / PostgreSQL
bcrypt (로컬 인증) / python-binance WebSocket / Jinja2 (SSR)
aiolimiter / MultiFernet / itsdangerous / Alembic
```

## 2. 루트 디렉토리 파일

### ✏️ `.env.example`
환경 변수 템플릿. 실제 운영 시 `.env` 파일로 복사하여 값을 채움.

주요 설정값:
- `DATABASE_URL`: PostgreSQL 직접 연결 주소 (트레이딩 엔진용)
- `ENCRYPTION_KEYS`: API 키 암호화용 Fernet 키 (쉼표 구분, 첫 번째가 최신)
- `SESSION_SECRET_KEY`: 세션 쿠키 서명 키
- `CSRF_SECRET`: CSRF 방어 토큰 시크릿
- `API_RATE_LIMIT`: 바이낸스 API 분당 최대 호출 가중치 (기본 1000)
- 관리자 생성: `python scripts/create_admin.py --email admin@example.com`

> ❌ 제거됨: `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY` (로컬 인증으로 전환)

### `pyproject.toml`
Python 프로젝트 메타데이터. 프로젝트 이름, 버전, Python 버전 요구사항 등 정의.

### `requirements.txt`
Python 패키지 의존성 목록. 주요 패키지:
- `fastapi`, `uvicorn`: 웹 프레임워크 + ASGI 서버
- `sqlalchemy[asyncio]`, `asyncpg`: 비동기 DB ORM
- `python-binance`: 바이낸스 API 클라이언트 (WebSocket 포함)
- `bcrypt`: 로컬 인증 비밀번호 해시
- `itsdangerous`: 세션 쿠키 서명
- `starlette-csrf`: CSRF 보호 미들웨어
- `aiolimiter`: 비동기 속도 제한기
- `cryptography`: MultiFernet 암호화
- `jinja2`: HTML 템플릿 엔진

### `.gitignore`
Git에서 무시할 파일 패턴. `.env`, `__pycache__`, `.venv` 등.

### `alembic.ini`
Alembic(DB 마이그레이션 도구) 설정 파일. DB 연결 문자열과 마이그레이션 스크립트 위치를 지정.

## 3. app/ - ✏️ 메인 애플리케이션

### `app/__init__.py`
빈 파일. `app` 디렉토리를 Python 패키지로 인식시킴.

### ✏️ `app/main.py` - FastAPI 앱 진입점
**프로젝트의 가장 핵심적인 파일.** 앱 시작/종료 전체 생명주기를 관리.

**역할:**
1. **FastAPI 앱 생성**: 제목, 버전, 라이프사이클 등록
2. **✏️ 라이프사이클(lifespan)**: 앱 시작 시 필요한 모든 서비스를 초기화
   - 로깅 설정 (`setup_logging`)
   - 스레드 풀 설정 (`ThreadPoolExecutor`)
   - 암호화 매니저 (`EncryptionManager`) 초기화
   - 인증 서비스 (`AuthService`) + 세션 매니저 (`SessionManager`) 초기화
   - **🆕 초기 관리자 계정**: `scripts/create_admin.py` 스크립트로 수동 생성
   - 트레이딩 엔진 (`TradingEngine`) 시작 - 모든 활성 계정의 자동매매 시작
   - **🆕 캔들 집계 백그라운드 태스크** (`CandleAggregator`) 시작
   - **🆕 고아 백테스트 정리** (`cleanup_orphan_backtests`): 시작 시 RUNNING 상태로 남은 백테스트 정리
3. **미들웨어 등록**:
   - `CSRFMiddleware`: SSR 폼 POST 요청에 CSRF 토큰 요구
   - **✏️ `LazyAuthMiddleware`**: 쿠키에서 로컬 DB 세션 토큰 추출 → `request.state.user`에 주입
4. **✏️ 라우터 등록**: 8개 API 라우터 + 1개 SSR 페이지 라우터 (`combos_router`, `backtest_router` 추가)
5. **정적 파일 서빙**: CSS, JS 파일을 `/static` 경로로 서빙

**✏️ `LazyAuthMiddleware`** 내부 동작:
- 공개 경로(`/health`, `/login`, `/api/auth/*`, `/static`)는 인증 없이 통과
- 나머지 경로: 쿠키에서 세션 토큰 추출 → **로컬 DB에서 uid/email/role 검증** → `request.state.user`에 주입
- 인증 실패 시: API 요청이면 401 반환, 페이지 요청이면 `/login`으로 리다이렉트

### ✏️ `app/config.py` - 글로벌 설정
`pydantic-settings`의 `BaseSettings`를 상속하여 `.env` 파일에서 환경 변수를 자동 로딩.

**✏️ 주요 필드:**
- `database_url`: PostgreSQL 직접 연결 (트레이딩 엔진용)
- `encryption_keys`: 쉼표 구분 다중 Fernet 키 문자열
- `session_secret_key`, `csrf_secret`: 세션/CSRF 시크릿
- `api_rate_limit`: 바이낸스 API 분당 가중치 한도 (기본 1000)
- `thread_pool_size`: `asyncio.to_thread` 풀 크기 (기본 20)
- 관리자 계정: `scripts/create_admin.py` 스크립트로 수동 생성

> ❌ 제거됨: `supabase_url`, `supabase_anon_key`, `supabase_service_role_key`

**특수 프로퍼티:**
- `encryption_key_list`: 쉼표로 구분된 `encryption_keys` 문자열을 리스트로 변환

### `app/dependencies.py` - FastAPI 의존성 주입
FastAPI의 `Depends()` 시스템에 사용되는 함수들. 라우터 핸들러가 필요한 객체를 자동으로 주입받음.

**제공하는 의존성:**
- `get_db()`: DB 세션 (SQLAlchemy AsyncSession)
- `get_trading_engine()`: 트레이딩 엔진 (app.state에서 가져옴)
- `get_current_user()`: 현재 인증된 사용자 (미인증 시 401)
- `require_admin()`: 관리자 권한 확인 (비관리자 시 403)
- `get_auth_service()`: 인증 서비스
- `get_session_manager()`: 세션 매니저
- `get_encryption()`: 암호화 매니저

## 4. ✏️ app/models/ - 데이터베이스 모델

SQLAlchemy 2.0의 선언적 매핑(`DeclarativeBase`)을 사용. 각 파일이 하나의 DB 테이블에 대응.

### `app/models/base.py` - 베이스 클래스
모든 모델이 상속하는 SQLAlchemy `Base` 클래스 정의.

### ✏️ `app/models/user.py` - 사용자 프로필 (`user_profiles` 테이블)
로컬 DB 인증으로 등록된 사용자 정보.

| 컬럼 | 타입 | 설명 |
|-------|------|------|
| `id` | UUID (PK) | 사용자 고유 ID |
| `email` | String | 이메일 (로그인 식별자) |
| `password_hash` | String | bcrypt 해시 비밀번호 |
| `display_name` | String (nullable) | 표시 이름 |
| `role` | String | 역할: `"user"` 또는 `"admin"` |
| `created_at` | Timestamp | 생성 시각 |

**관계:** `accounts` → 이 사용자가 소유한 트레이딩 계정 목록

### ✏️ `app/models/account.py` - 트레이딩 계정 (`trading_accounts` 테이블)
**시스템의 핵심 단위.** 하나의 바이낸스 API 키 + 매매 설정 묶음.

| 컬럼 | 타입 | 설명 |
|-------|------|------|
| `id` | UUID (PK) | 계정 고유 ID |
| `owner_id` | UUID (FK→user_profiles) | 소유자 |
| `name` | String | 계정 별명 (예: "메인계정", "테스트") |
| `exchange` | String | 거래소 종류 (기본: "binance") |
| `symbol` | String | 매매 대상 심볼 (예: "BTCUSDT", "ETHUSDT") |
| `base_asset` / `quote_asset` | String | 기초/인용 자산 (예: "BTC"/"USDT") |
| `api_key_encrypted` | String | **암호화된** 바이낸스 API 키 |
| `api_secret_encrypted` | String | **암호화된** 바이낸스 API 시크릿 |
| `encryption_key_version` | Integer | 암호화 키 버전 (키 로테이션용) |
| `is_active` | Boolean | 매매 활성 상태 |
| `circuit_breaker_failures` | Integer | 연속 실패 횟수 |
| `circuit_breaker_disabled_at` | Timestamp | 서킷 브레이커 발동 시각 |
| `last_success_at` | Timestamp | 마지막 성공 시각 |
| `loop_interval_sec` | Integer | 매매 루프 주기 (기본 60초) |
| `order_cooldown_sec` | Integer | 주문 간 최소 대기 (기본 7초) |
| **🆕** `buy_pause_state` | String (Enum) | Buy Pause 상태: `ACTIVE` / `THROTTLED` / `PAUSED` |
| **🆕** `buy_pause_reason` | String (nullable) | 일시정지 사유 |
| **🆕** `buy_pause_since` | Timestamp (nullable) | 일시정지 시작 시각 |
| **🆕** `consecutive_low_balance` | Integer | 연속 잔고 부족 횟수 |

**관계:** `strategy_states`, `orders`, `fills`, `lots`, `positions`, **🆕** `trading_combos`

**설계 포인트:**
- API 키는 **절대 평문 저장하지 않음**. MultiFernet으로 암호화 후 저장
- 서킷 브레이커: 연속 5회 실패 시 `is_active=false`로 전환, 관리자가 수동 리셋
- **🆕 `BuyPauseState` enum**: `ACTIVE` → `THROTTLED`(루프 지연) → `PAUSED`(매수 중단) 상태 기계

### 🆕 `app/models/trading_combo.py` - 트레이딩 콤보 (`trading_combos` 테이블)
Buy Logic + Sell Logic 조합 단위. 하나의 계정에 여러 콤보를 동시 운영 가능.

| 컬럼 | 타입 | 설명 |
|-------|------|------|
| `id` | UUID (PK) | 콤보 고유 ID |
| `account_id` | UUID (FK) | 소속 계정 |
| `name` | String | 콤보 이름 (예: "lot+fixedtp") |
| `buy_logic_name` | String | Buy Logic 이름 (예: "lot_stacking") |
| `sell_logic_name` | String | Sell Logic 이름 (예: "fixed_tp") |
| `buy_params` | JSONB | Buy Logic 파라미터 |

### `app/exchange/binance_client.py` - 바이낸스 실제 클라이언트
`ExchangeClient`를 구현한 실제 바이낸스 연동 클라이언트.

**핵심 설계: 동기 코어 + 비동기 래퍼**

`python-binance` 라이브러리는 동기(sync) 방식이므로, 모든 메서드를 `asyncio.to_thread`로 래핑하여 비동기 이벤트 루프를 블로킹하지 않음.

**심볼 필터 캐시:** `_filters_cache`로 한 번 조회한 심볼의 step_size/tick_size를 캐시하여 반복 API 호출 방지.

### `app/exchange/backtest_client.py` - 백테스트용 가상 클라이언트
`ExchangeClient`를 구현한 **시뮬레이션 클라이언트**. 실제 API 호출 없이 메모리에서 거래를 시뮬레이션.

**동작 방식:**
1. `set_price(price)` 호출로 현재 시뮬레이션 가격 설정
2. 가격 설정 시 자동으로 `_check_order_fills()` 호출 → 대기 중인 지정가 주문 체결 확인
3. 잔고, 주문, 체결 내역을 모두 메모리에서 관리

## 8. ✏️ app/services/ - 비즈니스 로직 서비스

### ✏️ `app/services/trading_engine.py` - 멀티 계정 트레이딩 엔진
**모든 계정의 자동매매를 총괄하는 오케스트레이터.**

**✏️ 동작 방식:**
1. `start()`: 모든 활성 계정을 DB에서 로드 → 각 계정마다 `AccountTrader`를 생성하고 asyncio Task로 실행
2. **스태거드 스케줄링**: 계정 간 시작 시간 분산
3. **✏️ `KlineWsManager` 관리**: 계정 심볼에 따라 WebSocket 구독/해제
4. `stop_account(id)`, `start_account(id)`: 개별 계정 시작/중지
5. `reload_account(id)`: 중지 후 재시작 (설정 변경 적용 시)
6. **✏️** `resume_buying(account_id)`: Buy Pause 상태에서 수동 재개
7. `get_account_health()`: 모든 계정의 건강 상태 조회

**공유 자원:**
- `PriceCollector`: 동일 심볼 가격 조회를 중복하지 않음
- `GlobalRateLimiter`: 모든 계정이 하나의 속도 제한기를 공유
- **🆕** `KlineWsManager`: WebSocket 기반 실시간 캔들 수신

### ✏️ `app/services/account_trader.py` - 단일 계정 매매 루프
**하나의 계정에 대한 자동매매 루프.** TradingEngine이 각 계정마다 하나씩 생성.

**✏️ `run_forever()` 메인 루프:**
```
while 실행중:
    try:
        step()을 호출하여 매매 사이클 1회 실행
    except 오류:
        연속 실패 카운터 증가 → 5회 이상이면 서킷 브레이커 발동
        아니면 → 지수 백오프 (1s, 2s, 4s, 8s, max 60s)
    성공 시:
        BuyPauseManager.compute_interval()로 루프 간격 동적 계산
        asyncio.Event 기반 인터럽트 가능한 sleep (재개 신호 즉시 반응)
```

**✏️ `step()` 매매 사이클:**
1. DB에서 계정 정보 로드
2. 레이트 리미터에서 API 호출 용량 확보
3. 거래소와 주문/체결 데이터 동기화
4. 현재 가격 조회 (PriceCollector 경유)
5. **✏️ 활성화된 콤보들의 Buy/Sell tick() 실행** (strategy_configs 대신 trading_combos 사용)
6. **✏️ BuyPauseManager.should_attempt_buy()** 확인 후 매수 진행 여부 결정
7. 성공 기록

### 🆕 `app/services/buy_pause_manager.py` - Buy Pause 매니저
잔고 부족 시 매수를 자동으로 일시정지하는 **상태 기계**.

**상태 전이:**
```
ACTIVE → THROTTLED (연속 잔고 부족 감지) → PAUSED (임계값 초과)
PAUSED → ACTIVE (수동 resume() 또는 잔고 회복)
```

**주요 메서드:**
- `should_attempt_buy(account_id)`: 현재 상태에서 매수 시도 가능 여부
- `update_state(account_id, had_sufficient_balance)`: 매매 사이클 후 상태 갱신
- `resume(account_id)`: 수동 재개 (`/buy-pause/resume` API에서 호출)
- `compute_interval(account_id)`: THROTTLED 상태일 때 루프 간격 연장 값 반환

### 🆕 `app/services/kline_ws_manager.py` - WebSocket 캔들 매니저

**주요 기능:**
- `subscribe(symbol)` / `unsubscribe(symbol)`: 레퍼카운트 기반 구독 관리

- **슈퍼바이저 루프**: 연결 끊김 시 자동 재연결
- **REST 백필**: 구독 시작 시 최근 캔들 REST API로 보충
- 수신된 1m 캔들을 `CandleStore`를 통해 DB에 저장

### 🆕 `app/services/candle_store.py` - 캔들 저장소
캔들 데이터 CRUD 서비스.

**주요 메서드:**
- `store_closed_candle_1m(symbol, candle_data)`: 1분봉 단건 저장
- `store_candles_batch_1m(symbol, candles)`: 1분봉 배치 저장 (REST 백필용)
- `get_candles(symbol, from_ms, to_ms, interval)`: 조회 (`price_repo` 위임)
- `aggregate_candles(symbol, from_interval, to_interval)`: 하위 타임프레임 → 상위 타임프레임 집계

### 🆕 `app/services/candle_aggregator.py` - 캔들 집계기
백그라운드에서 주기적으로 캔들을 집계하는 서비스. 앱 시작 시 `TradingEngine`이 시작.

- `loadAccounts()`: 계정 목록 로딩 및 카드 렌더링
- **✏️** `loadPriceChart(accountId, interval)`: 다중 타임프레임 캔들 차트 그리기
- `loadLots(accountId)`: 로트 테이블 렌더링 + 콤보별 필터 탭
- `loadTuneValues(accountId)`: 튜닝 폼에 현재 값 채우기
- `saveTuneValues(accountId)`: 튜닝 값 POST (CSRF 토큰 포함)
- `resetCircuitBreaker(accountId)`: 서킷 브레이커 리셋 API 호출
- **🆕** `renderBuyPauseBadge(state)`: THROTTLED(주황), PAUSED(노란) 배지 렌더링
- **🆕** `resumeBuying(accountId)`: Buy Pause 수동 재개 API 호출
- `loadAdminOverview()`: 관리자 페이지 데이터 로딩

## 20. ✏️ 데이터 흐름 요약

### ✏️ 매매 루프 (콤보 기반)
```
[TradingEngine]
    ├── [KlineWsManager] ──── WebSocket 1m 캔들 수신 → CandleStore 저장
    │
    ├── [AccountTrader #1] ──────────── asyncio Task
    │       │
    │       ├── [PriceCollector] ← WS 캐시 → REST 폴백
    │       ├── [GlobalRateLimiter] ← API 호출 속도 제한
    │       ├── [BuyPauseManager] ← 잔고 부족 여부 판단
    │       ├── [BinanceClient] ← 실제 거래소 API (asyncio.to_thread)
    │       │
    │       ├── [Combo #1: LotStackingBuy + FixedTpSell]
    │       │       ├── BaseBuyLogic.pre_tick() + tick()
    │       │       │       ├── StrategyStateStore (scope=combo_id_1)
    │       │       │       └── LotRepository
    │       │       └── BaseSellLogic.tick(buy_lots)
    │       │               └── ExchangeClient (익절 매도 주문)
    │       │
    │       └── [Combo #2: TrendBuy + FixedTpSell]
    │               └── StrategyStateStore (scope=combo_id_2) ← 격리된 상태
    │
    ├── [AccountTrader #2] ──────────── 독립 asyncio Task
    └── [AccountTrader #N] ──────────── ...
```

### 🆕 WebSocket 캔들 흐름
```
Binance WebSocket (kline stream)
    → KlineWsManager.multiplex_socket
        → 캔들 closed 이벤트
            → CandleStore.store_closed_candle_1m() → DB price_candles_1m
            → PriceCollector 캐시 갱신

CandleAggregator (6시간 주기)
    → 1m → 5m (7일) → DB price_candles_5m
    → 5m → 1h (30일) → DB price_candles_1h
    → 1h → 1d (90일) → DB price_candles_1d
```

### ✏️ 인증 흐름 (로컬 DB)
```
사용자 → /login → 이메일/비밀번호 입력
    → POST /api/auth/login
        → AuthService.authenticate(email, password)
            → user_profiles 테이블에서 bcrypt.verify()
        → SessionManager.create_session_cookie(uid, email, role)
            → itsdangerous: uid/email/role을 서명된 쿠키로 인코딩
        → Set-Cookie (httponly, secure, samesite=lax)
    → /accounts 리다이렉트

이후 모든 요청:
    LazyAuthMiddleware → 쿠키에서 세션 토큰 추출
    → itsdangerous 서명 검증 + 만료 확인
    → request.state.user에 주입
```

### ✏️ 보안 계층
```
1. 암호화: API 키 → MultiFernet 암호화 → DB 저장
2. 인증: 이메일/비밀번호 → bcrypt 검증 → 서명된 쿠키 세션 (로컬 DB)
3. 권한: owner_id 확인 (본인 계정만) + role 확인 (admin 전용 기능)
4. CSRF: starlette-csrf 미들웨어 → SSR POST 요청에 토큰 요구
5. 속도 제한: GlobalRateLimiter → 바이낸스 API 1000 weight/분
6. 장애 격리: 서킷 브레이커 → 5회 연속 실패 시 계정 자동 비활성화
7. 매수 보호: BuyPauseManager → 잔고 부족 시 자동 일시정지
```
